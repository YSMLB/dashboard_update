Пайплайн (как данные попадают на экран)

0) Коротко: что считается пайплайном в этом проекте
Пайплайн это цепочка:
Excel выгрузки из 1С и расписания -> синхронизация на сервер -> конвертация в JSON -> API -> UI.

Все вычисления в текущей версии работают поверх JSON файлов в public/ (без отдельной БД).
История хранится в отдельных *_history.json и используется для исторического режима.

Схемка высокого уровня

  1С/Excel + расписание            backend (Go)                 frontend (React)
          |                           |                               |
          v                           v                               v
    /mnt/oneC_share  ->  конвертеры -> public/*.json -> REST /api -> экраны/таблицы

1) Источники данных
1.1 1С/файловый сервер (сетевая папка)
Туда выгружаются:
ведомость.xls/.xlsx (пропуски и/или деталка)
студенты.xlsx (если ведётся отдельным файлом контингента)
РасписаниеНаДату.xls (расписание на конкретный день)

Смысл источников:
Ведомость используется для attendance.json, vedomost.json и students.json.
РасписаниеНаДату используется для schedule.json и schedule_history.json.
students.xlsx, если есть, “перепрошивает” контингент, чтобы список студентов был актуален.

1.2 Дополнительные источники (опционально)
Внутренний фетчер расписания с сайта ОКЭИ может использоваться как отдельный инструмент/задача,
если подключён в окружении и включён в сборку/обновление.

2) Доставка файлов на сервер
2.1 Монтирование шары 1С
Сетевая папка 1С монтируется на Linux (cifs) в локальный каталог, например /mnt/proba.
Это делается один раз на сервере администратором инфраструктуры.

2.2 Синхронизация в рабочую папку конвертеров
Бэкенд при обновлении подтягивает (синхронизирует) актуальные Excel в директорию конвертеров,
чтобы дальше работать уже с локальными файлами, а не с сетью.

Почему так:
конвертеры работают стабильнее на локальных файлах
меньше проблем с правами/доступом/временными разрывами сети
обновление можно делать атомарнее (сначала скопировать, потом конвертировать)

Схемка доставки

  \\1C\share  --(mount)-->  /mnt/proba  --(sync)-->  backend/internal/converter/*

3) Конвертация Excel -> JSON (ядро пайплайна)
3.1 Триггеры обновления данных
Есть два стандартных сценария:
автоматически по расписанию (cron внутри бэкенда, подходит для постоянной работы)
вручную через административный endpoint (подходит для “обновить сейчас” после новой выгрузки)

3.2 Последовательность RefreshData (Scheduler)
RefreshData это единая транзакция обновления “всего”.
Она строит/обновляет JSON в public/ по понятному набору шагов.

Шаг 1: мастер‑конвертер ведомости
Вход: ведомость.xls/.xlsx
Выход в public/:
students.json
attendance.json
attendance_history.json (накопление)
vedomost.json

Шаг 2: (опционально) отдельный контингент
Если присутствует файл студенты.xlsx, он переконвертирует students.json,
чтобы список студентов был максимально актуальным именно по контингенту.

Шаг 3: конвертер ежедневного расписания
Вход: РасписаниеНаДату.xls
Условие: students.json должен существовать (расписание разворачивается на студентов).
Выход в public/:
schedule.json (только выбранный день, обычно сегодня)
schedule_history.json (накопление)

Схемка последовательности (упрощённо)

  RefreshData
     |
     +--> ConvertMaster(ведомость) -> students + attendance + history + vedomost
     |
     +--> ConvertStudents(студенты.xlsx) [если есть] -> students.json
     |
     +--> ConvertDailySchedule(РасписаниеНаДату) [если есть students] -> schedule + schedule_history

3.3 Принципы накопления истории
attendance_history.json:
пополняется из attendance.json при каждом обновлении
используется для исторического режима и сверок за прошедшие даты

schedule_history.json:
пополняется из ежедневного расписания
при обновлении за конкретную дату старые записи этой даты заменяются новыми (чтобы исправления в расписании не копили мусор)

4) Хранилище данных для API
Все выходные JSON лежат в public/ и являются основной базой данных проекта на текущем этапе.
Исторические данные накапливаются в отдельных *_history.json.

Зачем такое хранилище:
быстро внедрить и разворачивать без БД
проще резервное копирование (бэкап файлов)
прозрачность: файлы можно открыть и посмотреть “как есть”

Ограничения:
масштабирование по нагрузке ограничено файловой моделью
при росте требований можно перейти на БД, сохранив текущие алгоритмы конвертации как ingest-слой

5) API слой (backend)
5.1 Что отдаёт API
Бэкенд поднимает HTTP API и отдаёт:
данные и агрегаты для главной (dashboard)
данные по посещаемости (summary, drilldown)
результаты сверки (reconcile): за день, за пару, детализация по группе
административные endpoints для ручного обновления/конвертации (под роль admin)

5.2 Главная связка данных для отображения
Расписание даёт planned (кто должен быть на занятиях).
Посещаемость даёт absent (у кого missed > 0).
Reconciliation собирает итог present/absent и раскладывает по отделениям/группам.

6) Фронтенд
6.1 Оперативный режим
использует “сегодня”
подтягивает сверку по дню и по парам
показывает проблемные отделения/группы и текущую картину

6.2 Исторический режим
пользователь выбирает дату
фронт запрашивает те же API, но с выбранной датой
можно провалиться до уровня отделение -> группа -> пара -> список студентов

Схемка пользовательского запроса (упрощённо)

  UI (выбор даты/пары)
          |
          v
   GET /api/...reconcile...
          |
          v
  backend читает public/*history.json -> считает агрегаты -> отдаёт JSON -> UI рисует

7) Деплой (упрощённая схема)
7.1 Сборка и раздача фронта
Статика фронтенда собирается в dist/ и отдаётся Nginx.

7.2 Проксирование API
Nginx проксирует /api на бэкенд (обычно localhost:8080).

7.3 Доступ к данным 1С
Бэкенд может работать в Docker контейнере и иметь доступ к примонтированной папке 1С (volume mount),
чтобы забирать новые выгрузки по расписанию.
