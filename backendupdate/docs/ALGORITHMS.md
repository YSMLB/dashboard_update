Алгоритмы (что и как считается)

0) Термины и контекст
Система строит два режима отображения:
Оперативный режим. Работа “сегодня”, данные обновляются автоматически.
Исторический режим. Просмотр любой даты из накопленной истории.

Два источника “истины”:
Расписание отвечает на вопрос “кто должен быть” (planned).
Посещаемость отвечает на вопрос “у кого есть пропуски” (absent по факту).

Ключевые артефакты:
students.json. Контингент: отделение → группа → список студентов.
schedule.json. Расписание на один день (обычно сегодня), развёрнутое на каждого студента.
schedule_history.json. Накопленная история расписания по дням.
attendance.json. Детальная посещаемость по датам (missed по студенту).
attendance_history.json. Накопленная история детальной посещаемости.
vedomost.json. Сводная ведомость по пропускам (агрегаты по студентам/группам/отделениям).

Схемка “что из чего считается”

  РасписаниеНаДату.xls    ведомость.xls/.xlsx
           |                     |
           v                     v
   schedule.json (+history)   attendance.json (+history)
           |                     |
           +----------+----------+
                      v
                Reconciliation
          (planned vs missed -> present/absent)

1) Алгоритм построения расписания на день (ConvertDailySchedule)
Вход: Excel “РасписаниеНаДату.xls/.xlsx”.
Выход: schedule.json (только выбранный день) и schedule_history.json (накопление).

1.1 Определение даты расписания
В некоторых версиях файла дата может находиться рядом с днём недели, в других — отсутствует.
Правило:
Если рядом с днём недели есть дата, она распознаётся и используется как “дата расписания”.
Если даты нет, используется текущая дата сервера.

Поддерживаемые варианты в одной строке:
Понедельник
Понедельник 17.03.2026
Понедельник 17.03.26
Понедельник 17.03

Нормализация даты:
Внутри создаются два формата:
DD.MM.YYYY (для человекочитаемой части и для префикса в schedule-структуре)
YYYY-MM-DD (для API и сверок)

1.2 Поиск “шапки” и определение колонок
Файл может иметь разные макеты:
Вариант A: col0 = “День недели”, col1 = “Номер пары”, группы начиная с col2
Вариант B: col0 = “Номер пары”, группы начиная с col1 (без отдельной колонки дня)

Алгоритм:
Сканируется первые ~20 строк, чтобы найти строку, похожую на шапку:
либо явно по словам “День недели”/“Номер пары”
либо эвристикой: в строке найдено минимум несколько (например 3+) кодов групп

После нахождения шапки:
Определяется колонка номера пары (lessonNumCol)
Определяется, с какой колонки начинаются группы (groupStartCol)
Дальше строится map: группа -> индекс колонки

1.3 Извлечение пар (дисциплина/преподаватель/номер пары)
Ключевая проблема Excel: объединённые и пустые ячейки.
Поэтому чтение делается не только из массива строк, а напрямую из файла по координатам:
Берём номер строки (Excel 1-based)
Строим имя колонки по индексу
Читаем значение ячейки по “A12”, “B12” и т.д.

Правила извлечения:
Если строка — “день недели” (и возможно дата), то пара может быть указана в этой же строке.
Если строка без дня недели, но с номером пары, берём её как строку данных пары.
Если номер пары “потерялся” из-за объединения, пробуем соседнюю колонку.

Дальше для каждой группы:
Если ячейка пустая — пары нет.
Если не пустая — разбирается содержимое на дисциплину и преподавателя.

1.4 Разворачивание расписания на контингент (students.json)
schedule формируется не “по группе”, а “по студенту”, чтобы дальше сверка могла считать planned по людям.

Правило:
Для каждой записи пары (группа + номер пары + дисциплина + преподаватель):
Берём список студентов этой группы из students.json
Для каждого студента добавляем запись LessonRecord:
date = “DD.MM.YYYY 0:00:00” (строка, дата в начале)
lessonNumber = 1..6
attendance = false (поле есть, но фактическая посещаемость берётся из attendance/vedomost)

Если группа есть в расписании, но её нет в students.json:
Записи для группы не создаются, так как неизвестно, кого разворачивать.

1.5 Накопление истории расписания (schedule_history.json)
schedule.json:
Перезаписывается каждый раз и содержит только один день (обычно сегодня).

schedule_history.json:
Хранит накопление по дням.
Алгоритм обновления:
1) Читаем текущую историю (если файла нет или сломан, создаём новую).
2) Удаляем все записи этой даты (по префиксу даты в строке rec.Date).
3) Добавляем новые записи этой даты.
4) Пересчитываем TotalGroups/TotalStudents.
5) Обновляем поле Period, расширяя диапазон дат по краям.

Схемка обновления schedule_history.json

  schedule_history.json (до)        новые записи за дату D
            |                                  |
            v                                  v
   удалить все записи с префиксом D     добавить записи за D
            |                                  |
            +---------------+------------------+
                            v
             пересчитать totals + обновить period

2) Алгоритм конвертации ведомости (ConvertMaster)
Вход: ведомость.xls/.xlsx (выгрузка из 1С).
Выход: students.json, attendance.json, vedomost.json и обновление attendance_history.json.

2.1 Поддержка xls и конвертация в xlsx
Если входной файл .xls:
Проверяем наличие уже подготовленного .xlsx рядом.
Если .xlsx отсутствует или старее .xls, запускаем конвертацию .xls -> .xlsx через Python скрипт.
Если конвертация не удалась, конвертер возвращает ошибку с подсказкой о необходимости конвертации.

2.2 Извлечение периода и даты по умолчанию
Во многих выгрузках 1С период указан в строке вида:
Параметры: ... Период: DD.MM.YYYY - DD.MM.YYYY ...

Алгоритм:
Ищем строку “Параметры:”
Вытаскиваем период (строку диапазона дат)
Левую дату периода используем как дату по умолчанию для детальной посещаемости,
если в детальных строках отсутствует явная дата.

2.3 Разбор структуры “умным” проходом по строкам
Проход идёт сверху вниз, при этом удерживаются текущие “контексты”:
currentDepartment
currentSpecialty
currentGroup

Распознаются типы строк:
строка отделения
строка специальности
строка группы (код группы приводится к нижнему регистру)
строка студента (номер в группе + ФИО)
строка детальной посещаемости (дата + missed)
строка сводной ведомости (total/bad/excused)

2.4 Генерация students.json (контингент)
Правило извлечения:
Если строка содержит номер в группе и ФИО (2–4 слова), это студент.
Он добавляется в структуру отделения/группы.

Дедуп контингента:
Внутри группы студент считается одинаковым по ФИО.
Повторы не добавляются.

2.5 Генерация attendance.json (детальная посещаемость)
Правило извлечения:
Если строка распознана как “посещаемость” и missed > 0, создаём/обновляем запись по студенту и дате.

Правило обновления:
Если у одного студента в выгрузке встречается одна и та же дата несколько раз,
missed суммируется, чтобы не терять часы.

Важно:
Если по студенту на дату missed = 0, записи может не быть. Это ок, это значит “нет пропусков”.

2.6 Генерация vedomost.json (сводная ведомость)
Поддерживаются два подхода:
Старый шаблон: значения берутся из фиксированных колонок (например “неуваж/уваж/всего”).
Новый/универсальный: в строке собираются все числовые значения (кроме первой колонки),
и “последнее число” трактуется как “всего”, предыдущие — как “уваж/неуваж” при наличии.

Fallback на агрегирование:
Если сводной ведомости в файле нет, но есть детальная attendance,
vedomost строится агрегированно из attendance:
по студенту суммируются пропуски, и эти часы считаются “неуважительными”,
потому что данных о причинах в детальной выгрузке нет.

2.7 Накопление истории посещаемости (attendance_history.json)
При каждом обновлении attendance.json дополнительно обновляется attendance_history.json:
Добавляются новые даты.
Если дата уже есть, записи за дату обновляются.

3) Алгоритм сверки “кто должен быть” vs “кто был” (ReconciliationService)
Цель: для заданной даты (и опционально пары) посчитать planned, present, absent и разложить по отделениям и группам.

3.1 Входы и формулы
planned:
берётся из расписания по дате (или по дате+пара).

absentSet:
строится по детальной посещаемости за дату: если missed > 0, студент попадает в absentSet.
Ключ идентификации: отделение|группа|студент.

present:
каждый planned студент считается присутствующим, если его нет в absentSet.

Итоги:
TotalPlanned = количество planned
TotalPresent = количество present
TotalAbsent = TotalPlanned - TotalPresent

3.2 Дедупликация planned
Сверка “за день”:
Студент учитывается один раз даже если у него несколько пар в этот день.

Сверка “за пару”:
Студент учитывается в рамках конкретной пары.

Детализация по группе на пару:
Дополнительно есть дедуп по ключу “отделение|группа|студент”, чтобы исключить дубликаты, если они пришли из расписания.

3.3 Группировка результата
Агрегация идёт по структуре:
итоги по дню
разбиение по отделениям
внутри отделения разбиение по группам

Для сверки по паре в групповой статистике дополнительно прикладываются:
discipline
teacher

3.4 Кэширование и TTL
Результаты ReconcileDay и ReconcileDayLesson кэшируются в памяти на 5 минут.
Ключи кэша:
date
date_lesson_N

Зачем это сделано:
Фронт может часто дергать одни и те же endpoints (перерисовки, фильтры, автообновление).
Кэш снимает нагрузку, не делая данные “слишком старыми”.

3.5 Поведение при отсутствии данных посещаемости
Если по выбранной дате нет ни одной записи посещаемости (детальной),
сверка возвращает пустой результат (0 planned/0 present/0 absent).
Зачем:
чтобы не показывать “псевдо‑присутствие” только на основании расписания, когда факта посещаемости на дату ещё нет.
