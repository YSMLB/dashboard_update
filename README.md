## Мониторинг посещаемости ОКЭИ

### 1. История и назначение проекта

Этот проект родился из реальной боли колледжа:  
руками выгружать ведомости из 1С, считать пропуски в Excel и пытаться понять, кто реально сидит на парах — невозможно.  

**Цель системы** — автоматически забирать данные из 1С и файлов расписания, превращать их в удобный JSON и показывать всем участникам процесса (администрации, отделениям, кураторам) понятную и наглядную картинку:

- **Оперативный режим** — что происходит **прямо сейчас**:  
  сколько студентов присутствует / отсутствует, по каким отделениям и группам есть проблемы, какие пары идут сейчас.
- **Исторический режим** — что происходило **в любой день в прошлом**:  
  статистика по дням, по парам, по отделениям и группам, динамика пропусков.

Система спроектирована так, чтобы:

- Работать **полностью автоматически**: забрать файлы с сетевого ресурса 1С → сконвертировать → показать на фронте.
- Быть **масштабируемой**: один и тот же бэкенд и фронт можно развернуть в другом колледже, просто поменяв точки монтирования и форматы файлов.
- Быть **понятной даже не‑программисту**: запуск и обновление сведены к набору простых шагов.

Проект уже обкатан на инфраструктуре ОКЭИ и готов к «массовому производству»: его можно тиражировать на другие учреждения с минимальными доработками.

### 2. Общая архитектура

Проект состоит из двух основных частей:

- **Backend (`backendupdate/backend`)**
  - Язык: Go
  - Основные задачи:
    - Забор файлов из 1С (через смонтированную сетевую шару).
    - Конвертация Excel‑файлов в нормализованные JSON‑структуры.
    - Формирование **исторической** и **оперативной** картины:
      - `attendance.json` / `attendance_history.json` — посещаемость.
      - `schedule.json` / `schedule_history.json` — расписание.
      - `vedomost.json`, `students.json` — сводные ведомости и контингент.
    - REST API для фронтенда (Gin): `/api/attendance/...`, `/api/dashboard/...`, `/api/admin/refresh-data` и т.д.
    - Планировщик (cron): регулярные обновления данных без вмешательства человека.

- **Frontend (`src/`)**
  - React + TypeScript
  - Современный UI (карточки, таблицы, графики) под большие мониторы и телевизоры.
  - Два основных режима:
    - **Оперативный** — текущий день.
    - **Исторический** — выбор даты и детализация до уровня отделения / группы / пары.

Сервер крутится в Docker‑контейнере, статику фронтенда отдаёт Nginx.  
Схема деплоя:

```text
Пользователь → Nginx → backend (Docker, порт 8080) → JSON‑файлы и 1С‑шара
```

### 3. Поток данных «от 1С до экрана»

1. **1С / файловый сервер**
   - Папка шары, например: `\\1C01\proba`
   - Сюда регулярно выгружаются:
     - `ведомость.xls` / `ведомость.xlsx` — сводная ведомость по пропускам.
     - `студенты.xlsx` (или её аналог) — контингент.
     - `РасписаниеНаДату.xls` — **расписание на один конкретный день** (обновляется каждый день).

2. **Монтирование шары на Linux‑сервере**
   - Шара монтируется в каталог, например `/mnt/proba`:
   - В `backend.env` задаётся:
     - `ONEC_SOURCE_DIR=/mnt/proba`

3. **Копирование файлов в директорию конвертеров**
   - При запуске/обновлении бэкенда вызывается:
     - `utils.SyncFromOneC(ONEC_SOURCE_DIR, backend/internal/converter)`
   - Результат:
     - В `backend/internal/converter` появляются актуальные:
       - `ведомость.xls` / `ведомость.xlsx`
       - `студенты.xlsx`
       - `РасписаниеНаДату.xls`

4. **Мастер‑конвертер ведомости**
   - Файл: `backend/internal/converter/master.go`
   - Вызывается через `Scheduler.RefreshData()`:
   - Делает:
     - `attendance.json` — деталка по дням (кто сколько пропустил).
     - `attendance_history.json` — **накопительная** история посещаемости.
     - `vedomost.json` — сводка по отделениям/специальностям.
     - `students.json` — контингент (отделения → группы → студенты).

5. **Конвертер расписания на дату**
   - Файл: `backend/internal/converter/daily_schedule.go`
   - Вход: `РасписаниеНаДату.xls` (один день).
   - Выход:
     - `schedule.json` — **только текущий день** (оперативный режим).
     - `schedule_history.json` — **накопительная история** расписания, по дням и парам.

6. **REST‑API**
   - `/api/attendance/reconcile/day?date=YYYY-MM-DD` — сводка по дню.
   - `/api/attendance/reconcile/day?date=...&lesson=N` — по конкретной паре.
   - `/api/attendance/summary` — общая статистика.
   - `/api/dashboard/...` — данные для главной страницы (карточки, графики).
   - `/api/admin/refresh-data` — ручной запуск обновления (по кнопке или по cron).

7. **Frontend**
   - **Оперативный режим**
     - Берёт сегодняшнюю дату.
     - Тянет reconcile по дню и по парам.
     - Показывает:
       - Карточки «сколько присутствует / отсутствует сейчас».
       - Таблицу по отделениям.
       - Таблицу по парам (кликабельно до групп).
   - **Исторический режим**
     - Пользователь выбирает дату в календаре.
     - Фронт запрашивает те же API, но с выбранной датой.
     - Можно провалиться до отделения / группы / пары за любую прошедшую дату, где есть и расписание, и посещаемость.

### 4. Локальный запуск (для разработчика)

#### 4.1. Предварительные требования

- Node.js (LTS 18+)
- Go (>= 1.24)
- Docker (для проверки контейнера, опционально)
- Python 3 (для конвертации старых `.xls` → `.xlsx`)

#### 4.2. Установка зависимостей фронтенда

```bash
cd /path/to/dashboardoksei-feat-history-fixes
npm install
```

#### 4.3. Запуск бэкенда без Docker

```bash
cd backendupdate/backend
go run ./cmd/server
```

Что произойдёт при старте:

- Загрузится конфиг (`config.Load()`), найдётся `public/` и директория `backend/internal/converter`.
- Настроится `Scheduler` и cron‑обновление.
- Вызовется первоначальное обновление:
  - `SyncFromOneC` (если настроен `ONEC_SOURCE_DIR`).
  - `RefreshData()` — соберёт все JSON‑файлы в `public/`.
- HTTP‑сервер поднимется на `http://localhost:8080`.

#### 4.4. Запуск фронтенда в dev‑режиме

```bash
cd /path/to/dashboardoksei-feat-history-fixes
npm run dev
```

По умолчанию фронт будет доступен на `http://localhost:5173` и проксировать API на `http://localhost:8080/api/...` (через настройки Vite/прокси).

### 5. Деплой на сервер (типичный сценарий)

#### 5.1. Подготовка сервера

- Linux (Debian/Ubuntu)
- Установить:
  - `docker`, `docker-compose` (опционально).
  - `nginx`.
  - `cifs-utils` (для монтирования шары 1С).

#### 5.2. Клонирование проекта и сборка

```bash
cd /home/USER/dashboard
git clone <repo_url> dashboard
cd dashboard/backendupdate
docker build -t dashboard-backend .
```

#### 5.3. Монтирование шары 1С

```bash
sudo mkdir -p /mnt/proba
sudo umount /mnt/proba 2>/dev/null || true
sudo mount -t cifs //1C01/proba /mnt/proba \
  -o username=<логин>,password='<пароль>',iocharset=utf8,vers=3.0
```

#### 5.4. Файл окружения для бэкенда

`/home/USER/dashboard/backend.env`:

```env
ONEC_SOURCE_DIR=/mnt/proba
OKSEI_SCHEDULE_URL=https://oksei.ru/studentu/raspisanie_uchebnykh_zanyatij
SERVER_HOST=0.0.0.0
SERVER_PORT=8080
JWT_SECRET=change-me-in-production
APP_ENV=production
```

#### 5.5. Запуск контейнера бэкенда

```bash
cd /home/USER/dashboard/backendupdate
docker rm -f dashboard-backend 2>/dev/null || true
docker run -d \
  --name dashboard-backend \
  --restart unless-stopped \
  --env-file /home/USER/dashboard/backend.env \
  -v /mnt/proba:/mnt/proba \
  -p 127.0.0.1:8080:8080 \
  dashboard-backend
```

Проверка:

```bash
curl http://127.0.0.1:8080/api/health
```

Ожидаемый ответ:

```json
{"service":"dashboard-backend","status":"ok","server_time":"..."}
```

#### 5.6. Сборка фронтенда и Nginx

1. Сборка фронта:

   ```bash
   cd /home/USER/dashboard
   npm install
   npm run build
   ```

   В результате появится папка `dist/` со статикой.

2. Nginx‑конфиг (примерно):

   ```nginx
   server {
       listen 80 default_server;
       server_name _;

       root /home/USER/dashboard/dist;
       index index.html;

       location /api/ {
           proxy_pass http://127.0.0.1:8080/api/;
           proxy_set_header Host $host;
           proxy_set_header X-Real-IP $remote_addr;
           proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
       }

       location / {
           try_files $uri /index.html;
       }
   }
   ```

3. Перезапуск Nginx:

   ```bash
   sudo nginx -t
   sudo systemctl reload nginx
   ```

### 6. Как адаптировать проект под другой колледж

Чтобы перенести систему в другое учреждение, нужно пройти по чек‑листу:

1. **Формат файлов из 1С**
   - Ведомость:
     - Структура листа: отделения, группы, студенты, даты по колонкам.
     - Если формат отличается — правим конвертер в `converter/attendance.go` / `converter/master.go`.
   - Расписание:
     - Аналог `РасписаниеНаДату.xls` — таблица «день недели / пара / группы по столбцам».
     - Если структура другая — адаптируем парсер в `converter/daily_schedule.go`.

2. **Сетевая шара**
   - Договориться с ИТ‑отделом: где будет лежать выгрузка 1С.
   - Прописать её в `ONEC_SOURCE_DIR` (например, `/mnt/college1c`).

3. **Коды отделений и групп**
   - В `students.json` и в расписании должны совпадать обозначения групп и отделений.
   - При необходимости добавить/поменять маппинги в вспомогательных функциях (`departmentForGroup` и т.п.).

4. **Права доступа и безопасность**
   - Настроить свои `JWT_SECRET`, логины, роли.
   - Ограничить доступ к `/api/admin/*` только администраторам (уже реализовано на уровне middleware).

5. **Брендинг**
   - Заменить логотипы, подписи, название учреждения в футере.
   - Всё это делается на уровне фронтенда (React‑компоненты, тексты).

### 7. Типовые сценарии использования

1. **Дежурный администратор**
   - Открывает «Оперативный» режим.
   - Видит число присутствующих/отсутствующих сейчас.
   - Кликает по таблице по парам → видит проблемные группы/отделения → может сразу связаться с ответственными.

2. **Заведующий отделением**
   - Открывает «Исторический» режим.
   - Выбирает дату.
   - Смотрит распределение пропусков по отделениям.
   - Проваливается в своё отделение → группы → отдельные пары.

3. **Аналитика/отчётность**
   - На основе `attendance_history.json` и `schedule_history.json` можно строить дополнительные отчёты:
     - динамика посещаемости по неделям;
     - сравнение отделений;
     - проблемные пары/преподаватели/группы и т.п.

### 8. Что важно понимать «самому дальнему пользователю»

- **Ничего руками в Excel считать не нужно.**
  - Ваша задача — чтобы 1С регулярно выгружала корректные файлы (`ведомость`, `РасписаниеНаДату`, `контингент`) в нужную папку.
  - Дальше всё делает сервер.

- **Если в интерфейсе везде «0 из 0», значит:**
  - либо за эту дату ещё **нет ведомости** (нет детальной посещаемости),
  - либо **нет расписания** на эту дату.

- **При появлении новых файлов:**
  - можно либо дождаться автоматического обновления по расписанию (cron),
  - либо руками запустить `/api/admin/refresh-data` (кнопка в админке / команда).

- Архитектура уже рассчитана на:
  - интеграцию с 1С через файловый обмен;
  - перенос на другие организации (параметризуемое расположение файлов и шары);
  - работу в закрытых сетях (всё внутри локальной инфраструктуры).
- Есть:
  - понятный REST‑API;
  - разделение на оперативный и исторический режимы;
  - отлаженный пайплайн конвертации Excel → JSON.
- Документация (этот файл) описывает:
  - как развернуть систему с нуля;
  - куда положить файлы;
  - как адаптировать под другие форматы и инфраструктуру.

Проект готов к масштабированию: его можно развернуть в любом колледже, где есть 1С и выгрузка ведомостей/расписания в Excel, а также Linux‑сервер с Docker и Nginx.
